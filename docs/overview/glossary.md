<div id="imported-glossary"><section id="glossary"> <p>This glossary is divided in two sections, the first one concerns Tezos, and the second one concerns the <a className="reference internal" href="#economic-protocol">economic protocol</a>. The definitions in the latter section may be different for other protocol versions.</p><section id="tezos"><h2>Tezos<a className="hash-link" href="#tezos" title="Direct link to Tezos" aria-label="Direct link to Tezos"></a></h2><dl><dt><span className="target" id="block">Block</span></dt><dd><p>The Tezos blockchain is a linked list of blocks (or actually, a tree when several competing branches exist). Blocks conceptually contain a header and a list of <a className="reference internal" href="#operation">operation</a>s, which are specific to the <a className="reference internal" href="#economic-protocol">economic protocol</a>.</p><p>The header itself decomposes into a <a className="reference internal" href="https://tezos.gitlab.io/shell/validation.html#shell-header" target="_blank"><span className="std std-ref">shell header</span></a> (common to all Tezos economic protocols), and a protocol-specific header. The shell header contains protocol-agnostic data such as the predecessor’s block hash and the block’s timestamp.</p></dd></dl><span id="def-context"></span><dl id="def-context-paris"><dt><span className="target" id="context">Context</span></dt><dd><p>The state of the blockchain. The context is defined by the <a className="reference internal" href="#economic-protocol">economic protocol</a> and typically includes information such as “this <a className="reference internal" href="#account">account</a> is credited with this many tez” and “this is the code for that <a className="reference internal" href="#smart-contract">smart contract</a>.”</p><p>The context is modified by <a className="reference internal" href="#operation">operation</a>s. For example, an <a className="reference internal" href="#operation">operation</a> can transfer tez from one <a className="reference internal" href="#account">account</a> to another, which modifies the part of the context that tracks <a className="reference internal" href="#account">account</a> credit.</p></dd><dt><span className="target" id="economic-protocol">Economic protocol</span></dt><dd><p>The economic protocol is the set of rules defining valid <a className="reference internal" href="#operation">operation</a>s and <a className="reference internal" href="#block">block</a>s, how the network agrees on the next block to build (the consensus algorithm), and how operations update the blockchain state, also called <a className="reference internal" href="#context">context</a>.</p><p>In Tezos, the economic protocol can be upgraded without interruption or forking of the blockchain. This is because the procedure for an upgrade is also defined within the economic protocol, which can thus update itself.</p></dd><dt><span className="target" id="fitness">Fitness</span> (a.k.a. score, weight)</dt><dd><p>For each block, the consensus algotrithm can compute a score called fitness which determines the quality of the chain leading to that block. The shell changes the head of the chain to the valid block that has the highest fitness.</p></dd><dt><span className="target" id="height">Height</span></dt><dd><p>See <a className="reference internal" href="#level">level</a>.</p></dd></dl><span id="def-level"></span><dl className="simple" id="def-level-paris"><dt><span className="target" id="level">Level</span> (a.k.a. block height)</dt><dd><p>The position of a <a className="reference internal" href="#block">block</a> in the blockchain, that is, the number of blocks since the genesis block, where the genesis block is at level 0.</p></dd><dt><span className="target" id="mempool">Mempool</span></dt><dd><p>A pool (set) of <a className="reference internal" href="#operation">operation</a>s maintained by a <a className="reference internal" href="#node">node</a> and not yet included in a <a className="reference internal" href="#block">block</a>.</p></dd></dl><span id="def-metadata"></span><dl id="def-metadata-paris"><dt><span className="target" id="metadata">Metadata</span></dt><dd><p>A (block or operation) metadata is a piece of data computed as a result of the application of the block or operation on an associated <a className="reference internal" href="#context">context</a>. The metadata consists of many pieces of information such as the operation receipts, rewards updates, voting period, etc.</p><p>A block’s metadata is the collections of operations metadata for all the operations included in the block (if the validation was successful).</p><p>For a detailed metadata content check the <a className="reference internal" href="https://tezos.gitlab.io/active/rpc.html" target="_blank"><span className="doc">Paris RPCs - Reference</span></a> under the prefix <code className="docutils literal notranslate"><span className="pre">../&lt;block_id&gt;/metadata</span></code>.</p></dd><dt><span className="target" id="node">Node</span></dt><dd><p>A peer in the P2P network. It maintains a local state and propagates <a className="reference internal" href="#block">block</a>s and <a className="reference internal" href="#operation">operation</a>s.</p></dd><dt><span className="target" id="operation">Operation</span></dt><dd><p>An operation transforms the <a className="reference internal" href="#context">context</a>; this is what makes the state of the chain change. Operations are grouped into <a className="reference internal" href="#block">block</a>s; thus, the chain progresses in batches. For the different kinds of operations defined by the protocol, see <a className="reference internal" href="#operation-kinds">operation kinds</a>.</p></dd><dt><span className="target" id="score">Score</span></dt><dd><p>See <a className="reference internal" href="#fitness">fitness</a>.</p></dd><dt><span className="target" id="shell">Shell</span></dt><dd><p>The shell is a software component of the <a className="reference internal" href="#node">node</a>. It is parameterized by a specific <a className="reference internal" href="#economic-protocol">economic protocol</a>. It serves as the bridge between the P2P layer (handling communication between nodes) and the <a className="reference internal" href="#economic-protocol">economic protocol</a> layer (handling the <a className="reference internal" href="#context">context</a>, <a className="reference internal" href="#operation">operation</a> application, scoring, etc.).</p></dd><dt><span className="target" id="weight">Weight</span></dt><dd><p>See <a className="reference internal" href="#fitness">fitness</a>.</p></dd></dl></section><section id="protocol"><h2>Protocol<a className="hash-link" href="#protocol" title="Direct link to Protocol" aria-label="Direct link to Protocol"></a></h2><span id="def-accuser"></span><dl id="def-accuser-paris"><dt><span className="target" id="accuser">Accuser</span></dt><dd><p>When a <a className="reference internal" href="#delegate">delegate</a> attempts <a className="reference internal" href="#double-signing">double signing</a> (or when it tries to abuse the network in another similar way), another <a className="reference internal" href="#delegate">delegate</a> can make an accusation, by providing evidence of the offense. The <a className="reference internal" href="#delegate">delegate</a> injecting the accusation in a newly baked block is called the accuser.</p><p>The accuser is awarded some funds from the security deposit of the accused.</p><p>When using <a className="reference internal" href="https://tezos.gitlab.io/introduction/tezos.html#octez" target="_blank"><span className="std std-ref">Octez</span></a>, accusation operations are emitted by the accuser daemon. Note that this daemon is not associated to a delegate: accusation operations are anonymous, and any delegate can include them in a block.</p></dd></dl><span id="def-account"></span><dl id="def-account-paris"><dt><span className="target" id="account">Account</span></dt><dd><p>An account is an address managed by the protocol. In the <a className="reference internal" href="#context">context</a>, each account is associated with a balance (an amount of tez available).</p><p>An account can be either an <a className="reference internal" href="#originated-account">originated account</a> or an <a className="reference internal" href="#implicit-account">implicit account</a>.</p></dd><dt><span className="target" id="baker">Baker</span></dt><dd><p>When a <a className="reference internal" href="#delegate">delegate</a> creates a new <a className="reference internal" href="#block">block</a>, it is called the baker of this block. <a className="reference internal" href="#baking-rights">Baking rights</a> are distributed to different delegates based on their available <a className="reference internal" href="#stake">stake</a>. Only a delegate with baking rights is allowed to bake. The baker selects transactions from the <a className="reference internal" href="#mempool">mempool</a> to be included in the block it bakes.</p><p>When using <a className="reference internal" href="https://tezos.gitlab.io/introduction/tezos.html#octez" target="_blank"><span className="std std-ref">Octez</span></a>, baking and other consensus actions are handled by the baker daemon, on behalf of one or more <a className="reference internal" href="#delegate">delegate</a> accounts. By extension, a baker designates the owner of such a delegate account, typically running the baker daemon on its behalf.</p></dd><dt><span className="target" id="baking">Baking</span></dt><dd><p>The act of creating a new <a className="reference internal" href="#block">block</a> by a <a className="reference internal" href="#baker">baker</a>.</p></dd><dt><span className="target" id="baking-rights">Baking rights</span></dt><dd><p><a className="reference internal" href="#baking">Baking</a>/<a className="reference internal" href="#attesting">attesting</a> a <a className="reference internal" href="#block">block</a> can only be done by a <a className="reference internal" href="#delegate">delegate</a> who holds the baking/attesting right for that block <a className="reference internal" href="#level">level</a> and <a className="reference internal" href="#round">round</a>. At the start of a <a className="reference internal" href="#cycle">cycle</a>, baking and attesting rights are computed for all the <a className="reference internal" href="#block">block</a> levels and rounds in the <a className="reference internal" href="#cycle">cycle</a>, based on the proportion of the <a className="reference internal" href="#stake">stake</a> of each <a className="reference internal" href="#delegate">delegate</a>.</p><p>For each <a className="reference internal" href="#block">block</a> level and <a className="reference internal" href="#round">round</a>, there is exactly one account that is allowed to bake, but several accounts are allowed to attest.</p></dd><dt><span className="target" id="burn">Burn</span></dt><dd><p>To ensure responsible use of the storage space on the public blockchain, there are some costs charged to users for consuming storage. These costs are burnt (i.e., the amount of tez is destroyed). For example, a per-byte storage cost is burnt for increasing the storage space of a smart contract; a fixed amount is burnt for allocating a new contract (which consumes space by storing its address on the blockchain).</p><p>See also <a className="reference internal" href="#fee">fee</a>.</p></dd><dt><span className="target" id="constant">Constant</span></dt><dd><p>Protocols are parameterized by several parameters called protocol constants, which may vary from one protocol to another or from one network to another.</p></dd><dt><span className="target" id="contract">Contract</span></dt><dd><p>See <a className="reference internal" href="#account">account</a>.</p></dd></dl><span id="def-cycle"></span><dl id="def-cycle-paris"><dt><span className="target" id="cycle">Cycle</span></dt><dd><p>A cycle is a sequence of consecutive <a className="reference internal" href="#block">block</a>s of fixed length (given by a protocol <a className="reference internal" href="#constant">constant</a>). E.g., cycle 12 started at block level 49152 and ended at <a className="reference internal" href="#block">block</a> level 53248.</p><p>Cycles are used as a unit of “time” in the <a className="reference internal" href="#block">block</a> chain. For example, the different phases in the amendment voting procedures are defined based on numbers of cycles.</p><p>The length of a cycle is a (parametric) protocol <a className="reference internal" href="#constant">constant</a>, and thus might change across different Tezos protocols.</p></dd></dl><span id="def-delegate"></span><dl className="simple" id="def-delegate-paris"><dt><span className="target" id="delegate">Delegate</span></dt><dd><p>An <a className="reference internal" href="#implicit-account">implicit account</a> that can participate in consensus and in governance. Actual participation is under further provisions, like having a <a className="reference internal" href="#minimal-stake">minimal stake</a>. An implicit account becomes a delegate by registering as such. Through <a className="reference internal" href="#delegation">delegation</a>, other accounts can delegate their rights to a delegate account. The delegate’s rights are calculated based on its <a className="reference internal" href="#stake">stake</a>. Note that <code className="docutils literal notranslate"><span className="pre">tz4</span></code> accounts cannot be delegates.</p></dd><dt><span className="target" id="delegation">Delegation</span></dt><dd><p>An <a className="reference internal" href="#operation">operation</a> in which an <a className="reference internal" href="#account">account</a> designates a <a className="reference internal" href="#delegate">delegate</a>. The delegating account’s balance increases the <a className="reference internal" href="#delegate">delegate</a>’s <a className="reference internal" href="#stake">stake</a> and consequently its <a className="reference internal" href="#baking-rights">baking rights</a> and <a className="reference internal" href="#attesting-rights">attesting rights</a>. However, the <a className="reference internal" href="#delegate">delegate</a> does not control the funds of the delegating <a className="reference internal" href="#account">account</a>, e.g., it can not spend them.</p></dd></dl><span id="def-double-signing"></span><dl className="simple" id="def-double-signing-paris"><dt><span className="target" id="double-signing">Double signing</span></dt><dd><p>The situation when a <a className="reference internal" href="#baker">baker</a> signs two different <a className="reference internal" href="#block">block</a>s at the same level and same round, is called double baking. Double baking is detrimental to the network and might be indicative of an attempt to double spend. The same goes for signing two different attestations at the same level and the same round. As such, double signing (i.e., double baking or double attesting) is punished by the network: an <a className="reference internal" href="#accuser">accuser</a> can provide proof of the double signing to be awarded part of the double signer’s deposit – see <a className="reference internal" href="https://tezos.gitlab.io/active/consensus.html#slashing-paris" target="_blank"><span className="std std-ref">Slashing</span></a>.</p></dd><dt><span className="target" id="failing-noop">Failing Noop</span></dt><dd><p>The <code className="docutils literal notranslate"><span className="pre">Failing_noop</span></code> operation implements a <em>No-op</em>, which always fails at <a className="reference internal" href="https://tezos.gitlab.io/active/validation.html#operation-validity-paris" target="_blank"><span className="std std-ref">application time</span></a>, and should never appear in <a className="reference internal" href="https://tezos.gitlab.io/active/validation.html#full-application-paris" target="_blank"><span className="std std-ref">applied blocks</span></a>. This operation allows end-users to <a className="reference internal" href="https://tezos.gitlab.io/active/blocks_ops.html#failing-noop-paris" target="_blank"><span className="std std-ref">sign arbitrary messages</span></a> which have no computational semantics.</p></dd><dt><span className="target" id="attesting">Attesting</span></dt><dd><p>When a <a className="reference internal" href="#block">block</a> is created and propagated on the network, delegates that have <a className="reference internal" href="#attesting-rights">attesting rights</a> for the matching block <a className="reference internal" href="#level">level</a> and <a className="reference internal" href="#round">round</a> can emit an attestation <a className="reference internal" href="#operation">operation</a>. Attestation operations are included in the next <a className="reference internal" href="#block">block</a>.</p></dd><dt><span className="target" id="attesting-rights">Attesting rights</span></dt><dd><p>See <a className="reference internal" href="#baking-rights">baking rights</a>.</p></dd></dl><span id="def-fee"></span><dl id="def-fee-paris"><dt><span className="target" id="fee">Fee</span></dt><dd><p>To ensure responsible use of computation resources of other nodes, and also to encourage active participation in the consensus protocol, users pay fees to bakers for including their <a className="reference internal" href="#operation">operation</a>s in <a className="reference internal" href="#block">block</a>s. For example, fees are paid to a baker for operations such as a <a className="reference internal" href="#transaction">transaction</a> or a revelation of a public key.</p><p>Currently, only <a className="reference internal" href="https://tezos.gitlab.io/active/blocks_ops.html#manager-operations-paris" target="_blank"><span className="std std-ref">manager operations</span></a>require collecting fees from its sender <a className="reference internal" href="#account">account</a>.</p><p>See also <a className="reference internal" href="#burn">burn</a>.</p></dd></dl><span id="def-gas"></span><dl className="simple" id="def-gas-paris"><dt><span className="target" id="gas">Gas</span></dt><dd><p>A measure of the number of elementary steps performed during the execution of a <a className="reference internal" href="#smart-contract">smart contract</a>. Gas is used to measure how much computing power is used to execute a <a className="reference internal" href="#smart-contract">smart contract</a>.</p></dd></dl><span id="def-implicit-account"></span><dl id="def-implicit-account-paris"><dt><span className="target" id="implicit-account">Implicit account</span></dt><dd><p>An <a className="reference internal" href="#account">account</a> that is linked to a public key. Contrary to a <a className="reference internal" href="#smart-contract">smart contract</a>, an implicit account cannot include a script and it cannot reject incoming transactions.</p><p>If <em>registered</em>, an <a className="reference internal" href="#implicit-account">implicit account</a> can act as a <a className="reference internal" href="#delegate">delegate</a>.</p><p>The address of an implicit account always starts with the letters <code className="docutils literal notranslate"><span className="pre">tz</span></code> followed by <code className="docutils literal notranslate"><span className="pre">1</span></code>, <code className="docutils literal notranslate"><span className="pre">2</span></code>, <code className="docutils literal notranslate"><span className="pre">3</span></code>, or <code className="docutils literal notranslate"><span className="pre">4</span></code> (depending on the signature scheme) and finally the hash of the public key. See <a className="reference internal" href="https://tezos.gitlab.io/active/accounts.html" target="_blank"><span className="doc">Accounts and addresses</span></a> for a more detailed explanation on addresses.</p></dd><dt><span className="target" id="layer-1">Layer 1</span></dt><dd><p>The primary blockchain i.e. the Tezos chain. Within any blockchain ecosystem, Layer 1 (L1) refers to the main chain to which side chains, rollups, or other protocols connect and settle to. The Layer 1 chain is deemed to be most secure, since it has the most value (or stake) tied to it, and be most decentralized and censorship resistant. However, transaction space is limited leading to low throughput and possibly high transaction costs. See <a className="reference internal" href="#layer-2">Layer 2</a>.</p></dd><dt><span className="target" id="layer-2">Layer 2</span></dt><dd><p>Layer 2 (L2) includes sidechains, rollups, payment channels, etc. that batch their transactions and write to the <a className="reference internal" href="#layer-1">Layer 1</a> chain. By processing transactions on layer 2 networks, greater scalability in speed and throughput can be achieved by the ecosystem overall, since the number of transactions the Layer 1 can process directly is limited. By cementing transactions from a L2 to L1, the security of the L1 chain backs those operations. Currently, Layer 2 solutions on Tezos are built as <a className="reference internal" href="#smart-rollup">smart rollup</a>s.</p></dd><dt><span className="target" id="michelson">Michelson</span></dt><dd><p>The built-in language used by a <a className="reference internal" href="#smart-contract">smart contract</a>.</p></dd></dl><span id="def-minimal-stake"></span><dl className="simple" id="def-minimal-stake-paris"><dt><span className="target" id="minimal-stake">Minimal stake</span></dt><dd><p>An amount of tez (e.g., 6000ꜩ) serving as a minimal amount for a delegate to have <a className="reference internal" href="#baking-rights">baking rights</a> and voting rights in a <a className="reference internal" href="#cycle">cycle</a>.</p></dd><dt><span className="target" id="operation-kinds">Operation kinds</span></dt><dd><p>The main kinds of operations in the protocol are transactions (to transfer funds or to execute smart contracts), accusations, activations, delegations, attestations, and originations. For the full list of operations, see <a className="reference internal" href="https://tezos.gitlab.io/active/blocks_ops.html" target="_blank"><span className="doc">Blocks and Operations</span></a>.</p></dd><dt><span className="target" id="originated-account">Originated account</span></dt><dd><p>See <a className="reference internal" href="#smart-contract">smart contract</a>.</p></dd></dl><span id="def-origination"></span><dl className="simple" id="def-origination-paris"><dt><span className="target" id="origination">Origination</span></dt><dd><p>A manager <a className="reference internal" href="#operation">operation</a> whose purpose is to create – that is, to deploy – a <a className="reference internal" href="#smart-contract">smart contract</a> on the Tezos blockchain.</p></dd><dt><span className="target" id="pvm">PVM</span></dt><dd><p>A PVM (Proof-generating Virtual Machine) is a reference implementation for a device on top of which a <a className="reference internal" href="#smart-rollup">smart rollup</a> can be executed. This reference implementation is part of the <a className="reference internal" href="#economic-protocol">economic protocol</a> and is the unique source of truth regarding the semantics of rollups. The PVM is able to produce proofs enforcing this truth. This ability is used during the final step of a <a className="reference internal" href="#refutation-game">refutation game</a>.</p></dd><dt><span className="target" id="refutation-game">Refutation game</span></dt><dd><p>A process by which the <a className="reference internal" href="#economic-protocol">economic protocol</a> solves a conflict between two <a className="reference internal" href="#rollup-committer">rollup committer</a>s. Note that the refutation mechanism used in Tezos <a className="reference internal" href="#smart-rollup">smart rollup</a>s corresponds to the notion of <a className="reference external" href="https://academy.binance.com/en/glossary/fraud-proof" target="_blank">fraud proofs</a> used in other blockchain/Layer 2 ecosystems.</p></dd><dt><span className="target" id="refutation-period">Refutation period</span></dt><dd><p>When the first <a className="reference internal" href="#rollup-commitment">rollup commitment</a> for a <a className="reference internal" href="#rollup-commitment-period">rollup commitment period</a> is published, a refutation period of two weeks starts to allow this commitment to be challenged.</p></dd><dt><span className="target" id="roll">Roll</span></dt><dd><p>deprecated; see <a className="reference internal" href="#minimal-stake">minimal stake</a>.</p></dd><dt><span className="target" id="rollup-commitment">Rollup commitment</span></dt><dd><p>A claim that the interpretation of all <a className="reference internal" href="#rollup-inbox">rollup inbox</a> messages published during a given period, and applied on the state of a parent rollup commitment, led to a given new state by performing a given number of execution steps of the <a className="reference internal" href="#pvm">PVM</a>.</p></dd><dt><span className="target" id="rollup-commitment-period">Rollup commitment period</span></dt><dd><p>A period of roughly 15 minutes during which all <a className="reference internal" href="#rollup-inbox">rollup inbox</a>messages must be processed by the <a className="reference internal" href="#rollup-node">rollup node</a> state to compute a <a className="reference internal" href="#rollup-commitment">rollup commitment</a>. A commitment must be published for each commitment period.</p></dd><dt><span className="target" id="rollup-committer">Rollup committer</span></dt><dd><p>An <a className="reference internal" href="#implicit-account">implicit account</a> that has published and made a deposit on a <a className="reference internal" href="#rollup-commitment">rollup commitment</a>.</p></dd><dt><span className="target" id="rollup-inbox">Rollup inbox</span></dt><dd><p>A sequence of messages from the Layer 1 to all the <a className="reference internal" href="#smart-rollup">smart rollup</a>s. The contents of the inbox are determined by the consensus of the <a className="reference internal" href="#economic-protocol">economic protocol</a>.</p></dd><dt><span className="target" id="rollup-node">Rollup node</span></dt><dd><p>A daemon required for deploying and operating <a className="reference internal" href="#smart-rollup">smart rollup</a>s. The rollup node is responsible for making the rollup progress by publishing <a className="reference internal" href="#rollup-commitment">rollup commitment</a>s and by playing <a className="reference internal" href="#refutation-game">refutation game</a>s.</p></dd><dt><span className="target" id="rollup-outbox">Rollup outbox</span></dt><dd><p>A sequence of messages from a <a className="reference internal" href="#smart-rollup">smart rollup</a> to the Layer 1. Messages are <a className="reference internal" href="#smart-contract">smart contract</a> calls, potentially containing tickets. These calls can be triggered only when the related <a className="reference internal" href="#rollup-commitment">rollup commitment</a> is cemented (hence, at least two weeks after the actual execution of the operation).</p></dd></dl><span id="def-round"></span><dl className="simple" id="def-round-paris"><dt><span className="target" id="round">Round</span></dt><dd><p>An attempt to reach consensus on a block at a given level. A round is represented by an index, starting with 0. Each round corresponds to a time span. A <a className="reference internal" href="#baker">baker</a> with <a className="reference internal" href="#baking-rights">baking rights</a> at a given round is only allowed to bake during the round’s corresponding time span. <a className="reference internal" href="#baking">Baking</a> outside of one’s designated round results in an invalid <a className="reference internal" href="#block">block</a>.</p></dd><dt><span className="target" id="smart-contract">Smart contract</span></dt><dd><p><a className="reference internal" href="#account">Account</a> which is associated to a <a className="reference internal" href="#michelson">Michelson</a> script. They are created with an explicit <a className="reference internal" href="#origination">origination</a> operation and are therefore sometimes called originated accounts. The address of a smart contract always starts with the letters <code className="docutils literal notranslate"><span className="pre">KT1</span></code>.</p></dd><dt><span className="target" id="smart-rollup">Smart Rollup</span></dt><dd><p>Smart rollups constitute a <a className="reference internal" href="#layer-2">Layer 2</a> solution that can be used to deploy either a general-purpose polyvalent Layer 2 blockchain (e.g., an EVM-compatible one), or an application-specific DApp. See <a className="reference internal" href="https://tezos.gitlab.io/active/smart_rollups.html" target="_blank"><span className="doc">Smart Optimistic Rollups</span></a>.</p></dd><dt><span className="target" id="stake">Stake</span></dt><dd><p>The amount of tokens that determines a <a className="reference internal" href="#delegate">delegate</a>’s weight in the governance process and in the selection of its baking and <a className="reference internal" href="#attesting-rights">attesting rights</a>. A delegate’s stake is usually given by the delegate’s own tokens plus the sum of tokens delegated to it. However, there are cases when this is not the case, see <a className="reference internal" href="https://tezos.gitlab.io/active/consensus.html#active-stake-paris" target="_blank"><span className="std std-ref">here</span></a> for details.</p></dd><dt><span className="target" id="staker">Staker</span></dt><dd><p>An <a className="reference internal" href="#implicit-account">implicit account</a> that made a security deposit.
